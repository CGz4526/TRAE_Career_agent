# 后端职途 · 2核2G 服务器与 gt-agent 并行部署可行性评估

> 评估基于本工作区实际代码与配置文件（非推测）。评估日期：2026-07-13
> 服务器：腾讯云轻量应用服务器 2核 / 2GB 内存，已运行 gt-agent（FastAPI + RapidOCR，占用约 500MB–1GB）

---

## 一、技术栈核实（读代码确认）

| 层 | 实际技术栈 | 证据文件 |
|----|-----------|---------|
| AI 服务（后端） | **FastAPI 0.115.6 + uvicorn 0.34（[standard]）** | `backend-python/requirements.txt` |
| Agent 框架 | langchain 0.3.14 / langchain-core / langchain-openai / langgraph 0.2.61 | 同上 |
| LLM 调用方式 | **外部 API**，默认 `ark.cn-beijing.volces.com`（火山方舟 doubao-pro-32k），**无本地大模型** | `backend-python/config.py` |
| 业务层（后端） | **Spring Boot 3.2.5（Java 17）**，mybatis-plus + Spring Security + WebSocket + WebFlux | `backend-java/pom.xml` |
| 数据库 | **MySQL 8.0**（mybatis-plus）+ **Redis**（spring-boot-starter-data-redis） | `docker-compose.yml`、`application.yml` |
| 前端 | **Vue 3 + Vite 5**，Element Plus + Monaco + Three.js；**已构建出静态 `dist/`** | `frontend/package.json`、`frontend/dist/` |
| 文档解析 | pdfplumber / pymupdf(PyMuPDF) / pillow（用于简历 PDF/图片解析，**瞬态 CPU，非常驻模型**） | `requirements.txt` |

**结论**：后端职途是「Python AI 服务 + Java 业务服务 + MySQL/Redis + 前端静态站」四件套。其中真正重的只有 Java + MySQL + Redis 这一组；Python AI 服务本身**比 gt-agent 轻得多**（gt-agent 有常驻 ~300MB 的 RapidOCR/Paddle 模型，本项目没有任何常驻模型）。

---

## 二、依赖与重型组件核查

**Python AI 服务依赖（关键判断）**：
- ✅ 仅调用外部 LLM API，依赖 `langchain-openai` 走 HTTP（httpx）。
- ❌ **无** torch / transformers / sentence-transformers / paddle / chroma / faiss / 本地 embedding / 本地向量库。
- ❌ **无** 本地 OCR 引擎（gt-agent 的 RapidOCR 不在本依赖里）。图片 OCR 走 `llm_vision_model`（外部视觉模型）。
- ⚠️ 仅有的 CPU 开销是 `pymupdf`/`pdfplumber` 解析上传的简历 PDF，属**瞬态**，解析完即释放。

> 也就是说：本项目的 Python 服务**不会**像 gt-agent 那样把一个大模型常驻在内存里，这是它能被塞进小内存服务器的关键。

**Java 业务服务**：
- 必须连 **MySQL（端口 3308）** 与 **Redis（端口 6380）**，否则无法完整启动业务（鉴权/JWT、数据持久化都依赖它们）。
- 通过 `WebClient` 调用 Python AI 服务（`ai-service.base-url`，默认 `http://localhost:8000`）。

**前端**：构建产物是纯静态文件，运行时**不消耗应用内存**，由 nginx 托管即可。

## 三、资源占用预估（基于代码 + 依赖常识）

| 组件 | 空闲/典型占用 | 极限限流后 | 说明 |
|------|--------------|-----------|------|
| Python AI 服务（uvicorn 单 worker，关 reload） | **350–500 MB** | ~350 MB | langchain/langgraph 导入成本约 250–350MB，无常驻模型；PDF 解析峰值 +100–200MB |

| gt-agent（已占用） | 500 MB – 1 GB | — | 平均约 700MB，OCR 高峰冲 1GB |

## 四、并行可行性计算

把「后端职途全栈」与「gt-agent」加总：

| 场景 | 后端职途 | + gt-agent(均) | + gt-agent(峰) | 与 2GB 关系 |
|------|---------|---------------|---------------|------------|
| 全栈·典型 | 1.1–1.5 GB | 1.8–2.2 GB | 2.1–2.5 GB | ❌ 超标/OOM |
| 全栈·极限限流 | ~1.1 GB | ~1.8 GB | ~2.1 GB | ⚠️ 临界，gt-agent 高峰即 OOM |
| **仅 Python AI + 前端**（丢 Java/MySQL/Redis） | ~0.4 GB | ~1.1 GB | ~1.4 GB | ✅ 有余量 |

**CPU**：2 核。gt-agent 的 Paddle OCR 是 CPU 密集；Java+MySQL+Python 在个人/低频访问下多为 I/O 等待（等 LLM API），CPU 竞争不致命，但高并发时会明显变慢。

## 五、明确结论

### 结论：**有条件可行** ⚠️

1. **后端职途「轻量核心」（Python AI 服务 + 前端静态）与 gt-agent 并行 → ✅ 可行**
   合计约 1.1–1.4 GB，内存有余量，是最稳的方案。代价：前端登录/注册走 Java，放弃 Java 则无法登录、无数据持久化（仅适合纯演示或改前端跳过鉴权）。

2. **后端职途「全栈」（Java + MySQL + Redis + Python + 前端）与 gt-agent 并行 → ❌ 不可靠**
   典型 1.8–2.2 GB，已超过 2GB；gt-agent OCR 高峰必触发 OOM Killer，随机杀进程。

3. **若必须全栈并行**，需满足以下**条件之一**才「可行」：
   - **(A) 升级服务器到 2核4G**（腾讯云轻量 4G 约多几十元/月）——最省心，全栈无忧；
   - **(B) 拆分重型组件**：把 MySQL/Redis 改用腾讯云托管数据库（TencentDB for MySQL / Redis），或把 Java 后端单独放另一台轻量实例；本地只跑 Python+前端+gt-agent；
   - **(C) 全栈 + 极限内存压控 + 接受风险**：严格按下方参数限内存，把总占用压到 ~1.8 GB，留 ~200MB 缓冲；gt-agent 若冲高仍有 OOM 风险，属「走钢丝」。

## 六、若采用「有条件可行」的部署方案（C 方案 / 轻量核心方案）

### 6.1 独立 venv 策略（关键：避免与 gt-agent 的 numpy/paddle 冲突）

> ⚠️ 仓库里 `backend-python/venv/` 是 **Windows venv（`Scripts/` 目录）**，在 Linux 服务器上**直接用不了**，必须新建。
> gt-agent 用 paddle/rapidocr，对 `numpy`/`pillow` 版本有特定要求；本项目虽大概率不硬依赖 numpy，但为彻底隔离，**务必用独立 venv，绝不 pip 装进系统 Python 或 gt-agent 的环境**。

```bash
# 在服务器上
python3 -m venv /opt/venvs/career-agent
source /opt/venvs/career-agent/bin/activate
pip install --upgrade pip
pip install -r /opt/app/backend-career-agent/backend-python/requirements.txt
```

（本项目的 numpy 由 pdfplumber/pymupdf 间接引入概率低；即便引入，独立 venv 也不会动到 gt-agent 的 paddle 环境。）

### 6.2 端口规划

| 服务 | 端口 | 说明 |
|------|------|------|
| gt-agent | 保持原端口不变 | 不动它，nginx 继续反代 |
| 后端职途 · Python AI | **8001** | `config.py` 改 `port=8001`，或 `.env` 注入 `PORT=8001` |
| 后端职途 · Java | **8080** | `application.yml` 已支持 `${PORT:8080}` |
| 后端职途 · MySQL | 3308（容器内）/ 不对外 | 仅本机 Java 访问 |
| 后端职途 · Redis | 6380（容器内）/ 不对外 | 仅本机 Java 访问 |
| nginx | 80 / 443 | 统一入口，按路径反代 |

> 生产环境务必在 `.env` 里设 `DEBUG=false`（即 `debug=false`），关闭 uvicorn `reload`，否则会多起一个 reloader 进程（白多吃 ~80–120MB）。

### 6.3 systemd service 示例

**Python AI 服务** `/etc/systemd/system/career-agent-python.service`：
```ini
[Unit]
Description=后端职途 Python AI 服务
After=network.target

[Service]
User=www-data
WorkingDirectory=/opt/app/backend-career-agent/backend-python
Environment=PATH=/opt/venvs/career-agent/bin
EnvironmentFile=/opt/app/backend-career-agent/backend-python/.env
ExecStart=/opt/venvs/career-agent/bin/python -m uvicorn main:app --host 127.0.0.1 --port 8001 --workers 1
# 内存上限（防止拖累 gt-agent）：极端方案可加 MemoryMax=700M
# MemoryMax=700M
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

**Java 服务** `/etc/systemd/system/career-agent-java.service`（需先在本地 `mvn package` 把 jar 传到服务器，避免服务器上跑 Maven 吃内存）：
```ini
[Unit]
Description=后端职途 Java 业务服务
After=network.target mysql.service redis.service

[Service]
User=www-data
WorkingDirectory=/opt/app/backend-career-agent/backend-java
Environment="JAVA_OPTS=-Xmx256m -Xss256k -XX:+UseG1GC -Dfile.encoding=UTF-8"
EnvironmentFile=/opt/app/backend-career-agent/backend-java/.env
ExecStart=/usr/bin/java $JAVA_OPTS -jar target/backend-java-0.1.0.jar
# 极限限流可加：MemoryMax=480M
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

> MySQL/Redis 若用 Docker：用 `docker-compose up -d` 起，并在 `my.cnf` 里加 `innodb_buffer_pool_size=128M`、`performance_schema=OFF`；Redis 加 `maxmemory 64mb`、`maxmemory-policy allkeys-lru`。若直接 apt 装：用 systemd 同样限 `MemoryMax`。

### 6.4 nginx 反代配置要点

前端 `dist/` 由 nginx 直接托管；API 按路径分流：
- `/api/` → Java `127.0.0.1:8080`（业务/鉴权）
- `/agent/` → Python `127.0.0.1:8001`（AI 功能）
- 静态资源 + SPA 回退到 `index.html`

```nginx
server {
    listen 80;
    server_name your.domain.com;   # 或先用 IP
    root /opt/app/backend-career-agent/frontend/dist;
    index index.html;

    # 前端静态（含 SPA history 回退）
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Java 业务服务
    location /api/ {
        proxy_pass http://127.0.0.1:8080/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Python AI 服务
    location /agent/ {
        proxy_pass http://127.0.0.1:8001/agent/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        # 模拟面试是流式/SSE，按需加：
        proxy_http_version 1.1;
        proxy_read_timeout 300s;
    }

    # gt-agent 保持原反代规则不变
}
```

> ⚠️ **前端构建时**要把 `VITE_API_BASE` 设为 `/api`、`VITE_AGENT_BASE` 设为 `/agent`（同源相对路径），重新 `vite build` 后再上传 `dist/`，nginx 才能正确分流。否则前端写死的是绝对域名，会绕开反代。

### 6.5 依赖隔离注意事项

1. **绝不共用 Python 环境**：gt-agent（paddle/rapidocr，锁特定 numpy/pillow）与本项目各用各的 venv，互不干扰。
2. **Java 与 Python 进程隔离**：二者通过 HTTP（WebClient / nginx）通信，不共享内存，天然隔离；只用 `JAVA_OPTS` 限 JVM 堆。
3. **MySQL/Redis 仅监听本机**：`bind-address=127.0.0.1`、Redis 不加密码也仅本机，避免暴露。
4. **统一用非 root 用户**（如 `www-data`）跑服务，systemd `User=` 指定。
5. **监控**：部署后 `free -h` / `ps aux --sort=-%mem` 观察，若持续 >1.9GB 立刻考虑升级 4G 或拆分。

## 八、补充：错开切换运行（最省内存的稳妥方案）

> 用户追问：不并行，而是**同一时刻只跑一个项目、用时再切换**，是否可行？

**结论：完全可行，且是 2核2G 上内存最安全的方案。**

### 8.1 为什么可行（内存账重算）

错开后，两个项目的峰值**永远不会重叠**，因此只需各自单独能塞进 2GB 即可：

| 单独运行 | 峰值占用 | 2GB 剩余 | 结论 |
|---------|---------|---------|------|
| gt-agent 单独（FastAPI + RapidOCR） | 0.5 – 1.0 GB | 1.0 – 1.5 GB | ✅ 宽松 |
| 后端职途全栈单独（Java+MySQL+Redis+Python+前端） | 1.1 – 1.5 GB | 0.5 – 0.9 GB | ✅ 放得下 |
| 后端职途轻量核心单独（Python+前端） | 0.4 – 0.7 GB | 1.3 – 1.6 GB | ✅ 宽松 |

> 关键点：**后端职途全栈单独跑也只占 ~1.5GB，不会 OOM**；gt-agent 单独跑也只占 ~1GB。两者不再叠加，OOM 风险彻底消失。CPU 也更宽裕——2 核全程给单个项目用。

### 8.2 切换机制（用 systemd + 切换脚本）

把每边服务分成「组」，切换时先停另一边再起本边。nginx 可常驻（见 8.3）。

`/usr/local/bin/switch-to-career.sh`（切到后端职途）：
```bash
#!/bin/bash
# 停 gt-agent 全套（请按你实际的 service 名替换 gt-agent.*）
sudo systemctl stop gt-agent.service gt-agent-nginx.service 2>/dev/null
# 起后端职途全套
sudo systemctl start career-agent-python.service career-agent-java.service \
                     mysql.service redis.service career-nginx.service
echo "已切换到 后端职途"
```

`/usr/local/bin/switch-to-gt.sh`（切回 gt-agent）：
```bash
#!/bin/bash
# 停后端职途全套
sudo systemctl stop career-agent-python.service career-agent-java.service \
                     mysql.service redis.service career-nginx.service 2>/dev/null
# 起 gt-agent
sudo systemctl start gt-agent.service gt-agent-nginx.service
echo "已切换到 gt-agent"
```
```bash
chmod +x /usr/local/bin/switch-to-*.sh
```

> 进阶（可选）：把每边做成一个 **systemd target** 并用 `Conflicts=` 互相排斥，`systemctl isolate career.target` 会自动停掉 gt.target。比脚本更"系统原生"，但脚本更直观好改，个人用脚本足够。

### 8.3 nginx 处理（常驻，按子域名分流）

让 nginx 一直跑（~20MB，可忽略），用**子域名**区分两个项目的前端，两个 server 块都常驻配置；没启动的那边后端返回 502，但你本就不会同时用，无影响：

```nginx
# gt-agent 前端
server {
    listen 80;
    server_name gt.your.domain;
    location / { proxy_pass http://127.0.0.1:<gt-agent-port>; }
}
# 后端职途 前端（静态）+ API
server {
    listen 80;
    server_name career.your.domain;
    root /opt/app/backend-career-agent/frontend/dist;
    location / { try_files $uri $uri/ /index.html; }
    location /api/   { proxy_pass http://127.0.0.1:8080/; }
    location /agent/ { proxy_pass http://127.0.0.1:8001/agent/; }
}
```
切换项目时**无需动 nginx**，只起停后端 service 即可。

### 8.4 折中方案（混合常驻，推荐个人用）

不想每次都切、又怕 OOM，可以**常驻轻量部分、按需起重型部分**：

- **常驻**：gt-agent（一直开）+ 后端职途「Python AI + 前端」（已证并行可行，~1.1–1.4GB）。
- **按需起停**：后端职途的 Java + MySQL + Redis（~0.8–1.0GB）。只在需要登录/账号/数据持久化时才 `systemctl start career-agent-java mysql redis`；用完 `stop` 把内存还給 gt-agent（尤其 OCR 高峰时）。

这样两个项目"大部分时间都能访问"，又不会在 gt-agent OCR 冲高时 OOM。

### 8.5 切换方案的代价

- **同时只能用一个项目**：业务互斥，个人用/演示足够，但无法两项目同时对外服务。
- **切换有启动延迟**：MySQL 冷启 ~10–30s、JVM ~5–10s，整套切过去约 30–60s 才完全就绪（脚本跑完不等于服务立刻可访问，需等健康检查的端口起来）。
- **数据不丢**：MySQL/Redis 用 volume/数据目录持久化，停了再起数据还在。

> 一句话：**错开切换是 2核2G 上最稳的部署形态，全栈都能跑，只是不能同屏在线。**
