## 1. 产品概述

### 1.1 产品定位

「后端职途」是一款面向 Java / Python 后端技术岗求职者的**多智能体协同求职工作台**。系统基于 LangGraph 编排多个专业化 AI Agent，覆盖从岗位分析、简历优化、题库生成、项目梳理到模拟面试的求职全链路，帮助求职者以工程化、系统化的方式提升面试竞争力。

产品以「Agent 即同事」为设计理念，将求职准备过程拆解为若干可独立调度的智能体任务，模拟一个由岗位分析师、简历顾问、出题官、项目教练和面试官组成的虚拟求职辅导团队，为用户产出结构化、可追溯、可复用的求职资产。

### 1.2 目标用户

| 维度 | 描述 |
|------|------|
| 职业方向 | Java 后端、Python 后端、Go 后端、全栈、算法、大数据、DevOps 等技术研发岗 |
| 经验阶段 | 应届校招生（0 经验）、1-3 年初级工程师、3-5 年中级工程师 |
| 核心诉求 | 精准解析 JD、定向优化简历、高效刷题、还原真实面试场景、量化面试表现 |
| 技术特征 | 具备基本编程能力，熟悉 Git / 命令行操作，期望用工程化思维管理求职流程 |

### 1.3 核心价值

1. **多 Agent 协同，覆盖求职全链路**：单一平台完成「JD 分析 → 简历优化 → 题库生成 → 项目梳理 → 模拟面试」闭环，避免在多个割裂工具间反复切换。
2. **结构化产出，告别模糊建议**：所有 Agent 输出均为 Pydantic 结构化数据（岗位画像、逐条修改建议、分类题库、STAR 项目描述、多维评分报告），可追溯、可对比、可复用。
3. **角色卡驱动，零门槛上手**：预设 8 个通用角色卡，用户无需先做岗位匹配即可直接使用题库生成、简历优化、模拟面试等任意模块，降低首次使用成本。
4. **专属角色卡，实现微调级个性化**：岗位匹配后从 JD 画像自动生成用户专属角色卡，后续所有 Agent 基于该画像工作，效果类似对通用大模型做了一次「岗位微调」。
5. **模块解耦，按需取用**：各功能模块独立运行，用户可根据当前求职阶段灵活组合，不强制走完全流程。

### 1.4 产品愿景

让每一位后端求职者都能拥有一支 7×24 小时在线的 AI 求职辅导团队，将求职准备从「凭感觉、靠运气」升级为「有数据、可量化、可迭代」的工程化过程。

## 2. 市场分析

### 2.1 痛点分析

| 痛点编号 | 痛点描述 | 影响人群 | 现有方案的不足 |
|----------|----------|----------|----------------|
| P1 | **JD 信息过载，关键要求难提取**：一篇 JD 动辄上千字，技术栈、经验、学历、软技能混杂，求职者难以快速判断自身匹配度 | 所有求职者 | 人工逐字阅读，耗时且易遗漏关键加分项 |
| P2 | **简历优化靠模板，缺乏针对性**：市面简历工具多为排版美化，无法针对具体岗位的技术栈和核心技能做定向优化 | 应届生、初级工程师 | 通用模板无法识别「该岗位看重高并发经验」等隐含要求 |
| P3 | **刷题无方向，八股文海战术**：LeetCode / 牛客题库庞大但缺乏岗位导向，求职者不知道该重点刷哪些题 | 1-3 年工程师 | 题库与岗位脱节，复习效率低下 |
| P4 | **项目经历讲不清，STAR 法则难落地**：知道自己做过项目，但无法用结构化方式表达「背景-任务-行动-成果」 | 应届生、转岗者 | 缺乏引导式梳理工具，简历项目描述流水账化 |
| P5 | **面试无实战环境，紧张导致发挥失常**：真实面试机会有限，缺乏低成本的模拟训练场景 | 所有求职者 | 找人模拟协调成本高，AI 对话工具缺乏专业面试逻辑 |
| P6 | **面试表现无量化反馈，不知如何改进**：面完只能凭感觉判断好坏，缺乏多维度评分和具体改进建议 | 所有求职者 | 无结构化评分报告，无法定位薄弱环节 |
| P7 | **求职资产分散，历史记录难复用**：JD、简历、题库、面试记录散落在不同工具中，无法关联追溯 | 多次求职者 | 工具割裂，无法形成可复用的求职知识库 |

### 2.2 竞品分析

| 竞品 | 定位 | 核心功能 | 优势 | 不足 |
|------|------|----------|------|------|
| **ChatGPT / 豆包等通用大模型** | 通用 AI 对话 | 自由问答、文本生成 | 模型能力强、覆盖面广 | 缺乏求职领域结构化流程，输出不可控、不可复用 |
| **牛客网** | 技术求职社区 | 题库、面经、在线笔试 | 题库丰富、社区活跃 | 无 JD 解析、无简历优化、无模拟面试评分 |
| **LeetCode** | 算法刷题平台 | 算法题、竞赛 | 算法题质量高 | 仅覆盖算法，无八股文、系统设计、项目梳理 |
| **超级简历 / 木及简历** | 简历制作工具 | 简历模板、排版优化 | 模板美观、操作简单 | 无岗位匹配分析，优化建议泛泛而谈 |
| **面试鸭** | 面试题库平台 | 八股文题库 | 题库分类清晰 | 无 JD 驱动出题，无模拟面试交互 |
| **Pramp / Exponent** | 模拟面试平台 | 真人对练、AI 面试 | 真实交互体验 | 真人模式需预约，AI 模式缺乏后端专业深度 |

### 2.3 差异化优势

1. **多 Agent 编排，流程化闭环**：区别于通用大模型的「单轮问答」，本系统通过 LangGraph 编排 5 个专业 Agent，形成可追踪状态、可并行执行、可条件路由的工作流，覆盖求职全链路。
2. **JD 驱动的定向优化**：所有 Agent 的输出均以「岗位需求画像」为锚点，简历优化、题库生成、模拟面试均针对具体岗位的技术栈和核心技能定制，而非泛泛而谈。
3. **角色卡系统，兼顾通用与专属**：8 个预设角色卡解决「零门槛上手」，专属角色卡实现「微调级个性化」，这是通用工具和单一功能平台都不具备的分层个性化能力。
4. **模块解耦，按需取用**：用户可独立使用任意模块（如直接生成 Java 题库），也可走完整流程（JD → 简历 → 题库 → 面试），灵活适配不同求职阶段。
5. **结构化评分，量化面试表现**：模拟面试产出 5 维度评分报告（技术深度、代码能力、表达能力、系统设计、综合素质），帮助用户精准定位薄弱环节。
6. **求职资产沉淀，历史可复用**：所有 JD、简历、题库、面试记录持久化存储并相互关联，形成个人求职知识库，支持多次求职周期复用。

## 4. 功能需求

### 4.1 功能模块总览

系统功能分为三大类：**已实现功能**（V1.0 交付）、**本期新增功能**（V2.0 规划）、**远期规划功能**（V3.0 储备）。下文按模块逐一列出功能点。

优先级定义：
- **P0（必须）**：核心路径功能，阻塞主流程，必须本期交付
- **P1（重要）**：显著提升体验，本期尽量交付
- **P2（可选）**：锦上添花，资源允许时交付

### 4.2 用户认证与账户模块

| 编号 | 功能名 | 优先级 | 状态 | 功能描述 | 验收标准 |
|------|--------|--------|------|----------|----------|
| U-01 | 用户注册 | P0 | 已实现 | 用户名 + 密码注册，密码 BCrypt 哈希存储 | 注册成功返回 JWT；重复用户名返回 400 错误提示 |
| U-02 | 用户登录 | P0 | 已实现 | 用户名 + 密码校验，签发 24h 有效 JWT | 登录成功返回 token 和用户信息；密码错误返回 401 |
| U-03 | JWT 鉴权 | P0 | 已实现 | 所有业务 API 经 JwtAuthenticationFilter 校验 | 无效/过期 token 返回 401；合法 token 注入 userId |
| U-04 | 用户信息查询 | P1 | 已实现 | 获取当前登录用户的基本信息和技术栈 | 返回用户名、邮箱、技术栈字段 |
| U-05 | 技术栈标签管理 | P1 | 新增 | 用户可编辑个人技术栈标签，用于技能差距分析基线 | 支持新增/删除技术标签；保存后立即生效 |

### 4.3 角色卡模块（新增）

| 编号 | 功能名 | 优先级 | 状态 | 功能描述 | 验收标准 |

| R-02 | 预设角色卡详情 | P0 | 新增 | 查看角色卡的完整数据（技术栈、核心技能、面试方向、推荐题库分类） | 详情页展示完整结构化数据 |

## 5. 角色卡系统设计

角色卡系统是 V2.0 的核心创新，旨在解决「冷启动门槛高」和「个性化不足」两大问题。通过预设通用角色卡 + 专属角色卡的分层设计，实现「零门槛上手」与「微调级个性化」的平衡。

### 5.1 设计理念

```
┌─────────────────────────────────────────────────────────┐
│                    角色卡分层架构                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   预设角色卡层（通用基座，开箱即用）                          │
│   ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│   │Java  │Python│ Go   │前端  │算法  │全栈  │大数据│DevOps│
│   │后端  │后端  │后端  │      │      │      │      │      │
│   └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
│          │  用户粘贴 JD → 岗位匹配 Agent 解析              │
│          ▼                                              │
│   专属角色卡层（JD 驱动，微调级个性化）                      │
│   ┌─────────────────────────────────────────────────┐   │
│   │  用户专属角色卡（含 JD 提取的技术栈/核心技能/方向）  │   │
│   └─────────────────────────────────────────────────┘   │
│          │  下游所有 Agent 基于活跃角色卡工作               │
│          ▼                                              │
│   ┌──────────┬──────────┬──────────┬──────────┐         │
│   │题库生成   │简历优化   │模拟面试   │差距分析   │         │
│   └──────────┴──────────┴──────────┴──────────┘         │
└─────────────────────────────────────────────────────────┘
```

**核心思路**：预设角色卡相当于「通用大模型」，覆盖常见技术方向；专属角色卡相当于「微调后的模型」，针对用户的具体目标岗位做了精调。下游 Agent 统一从「活跃角色卡」读取上下文，无需关心来源是预设还是专属。

### 5.2 角色卡数据结构

```python
class RoleCard(BaseModel):
    """角色卡统一数据结构"""

    # === 基础信息 ===
    card_id: str                    # 角色卡唯一标识
    card_type: str                  # "preset" 预设 | "custom" 专属
    role_name: str                  # 角色名称，如 "Java后端开发"
    icon: str                       # 图标标识（预设卡使用固定图标）
    description: str                # 角色一句话描述

    # === 技术画像 ===
    tech_stack: List[str]           # 技术栈列表，如 ["Java", "Spring Boot", "MySQL", "Redis"]
    core_skills: List[str]          # 核心技能要求，按重要度排序
    soft_skills: List[str]          # 软技能要求
    experience_years: int           # 经验要求年数（预设卡为建议值，专属卡为 JD 实际值）
    education: str                  # 学历要求

    # === 面试导向 ===
    interview_directions: List[str] # 面试重点方向，如 ["JVM原理", "并发编程", "Spring源码"]
    question_categories: List[str]  # 推荐题库分类权重，如 ["八股文:40%", "算法:20%", "系统设计:30%", "项目深挖:10%"]
    difficulty_profile: str         # 难度画像，如 "中等偏难"

    # === 来源元数据（专属卡专属） ===
    source_jd_id: Optional[int]     # 专属卡的来源 JD ID
    source_company: Optional[str]   # 专属卡的来源公司
    created_at: str                 # 创建时间

    # === 关联用户 ===
    user_id: Optional[int]          # 预设卡为 None（全局共享），专属卡关联用户
```

### 5.5 专属角色卡生成规则

岗位匹配 Agent 输出 `JobProfile` 后，系统按以下规则转换为专属 `RoleCard`：

| JobProfile 字段 | RoleCard 字段 | 转换规则 |
|-----------------|---------------|----------|
| position | role_name | 直接映射 |
| tech_stack | tech_stack | 直接映射 |
| core_skills | core_skills | 直接映射 |
| soft_skills | soft_skills | 直接映射 |
| experience_years | experience_years | 直接映射 |
| education | education | 直接映射 |
| key_responsibilities | interview_directions | LLM 提取关键词作为面试重点 |
| - | question_categories | 根据 tech_stack 匹配最接近的预设卡分类权重 |
| - | difficulty_profile | 根据 experience_years 推断（<2=中等，2-5=中等偏难，>5=困难） |
| - | card_type | 固定为 "custom" |
| - | source_jd_id | 关联 JD 记录 ID |

### 5.6 角色卡与 JobProfile 的关系

| 维度 | 预设角色卡 | JobProfile | 专属角色卡 |
|------|-----------|------------|-----------|
| 来源 | 人工预设，全局共享 | JD 解析实时产出 | 从 JobProfile 转换生成 |
| 持久化 | 代码/配置文件常量 | job_descriptions 表 | role_cards 表 |
| 作用域 | 所有用户共享 | 单次分析 | 单用户专属 |
| 字段丰富度 | 最全（含面试方向、分类权重） | 中等（JD 提取的 8 字段） | 较全（JobProfile + 推导字段） |
| 下游使用 | 题库/简历/面试/差距分析均可直接用 | 仅简历优化和题库生成可用 | 题库/简历/面试/差距分析均可直接用 |

**关键设计决策**：专属角色卡是 JobProfile 的「增强版」，在 JobProfile 基础上补充了面试方向、分类权重、难度画像等推导字段，使其能独立驱动所有下游 Agent，而非仅驱动简历优化和题库生成。

## 6. 模块解耦方案

### 6.1 解耦目标

V1.0 中所有 Agent 强依赖岗位匹配（必须先解析 JD 才能使用其他模块），导致用户冷启动成本高。V2.0 的解耦目标是：

1. **题库生成、简历优化、模拟面试均可独立运行**，不强制依赖岗位匹配
2. **统一以角色卡作为上下文入口**，无论预设卡还是专属卡均可驱动各模块
3. **保留全流程编排能力**，用户仍可一键执行 JD → 简历 + 题库 + 项目的完整流程

### 6.2 解耦前后对比

```
【V1.0 强耦合架构】
JD 文本 → 岗位匹配 Agent → JobProfile
                              │
                   ┌──────────┼──────────┐
                   ▼          ▼          ▼
               简历优化    题库生成    模拟面试
              （必须先有    （必须先有   （必须先有
               JobProfile）  JobProfile） JobProfile）

【V2.0 解耦架构】
                            ┌─ 预设角色卡（8个）
    活跃角色卡（统一入口） ◀─┤
                            └─ 专属角色卡（JD 生成）
                                    │
              ┌─────────┬───────────┼───────────┬──────────┐
              ▼         ▼           ▼           ▼          ▼
          题库生成   简历优化    模拟面试    项目梳理    差距分析
         （独立运行）（独立运行）（独立运行）（独立运行）（独立运行）
              │         │           │           │          │
              └─────────┴───────────┴───────────┴──────────┘
                                    │
                            （可选）全流程编排
                        JD → 专属角色卡 → 并行执行各模块
```

### 6.3 各模块独立运行方案

#### 6.3.1 题库生成模块

| 维度 | V1.0（耦合） | V2.0（解耦） |
|------|-------------|-------------|
| 输入 | JobProfile（必须先解析 JD） | 活跃角色卡（预设或专属） |
| 调用方式 | `generate_questions(profile: JobProfile)` | `generate_questions(card: RoleCard)` |
| 兼容策略 | - | 内部将 RoleCard 转换为 JobProfile 兼容结构后复用原有 Agent 逻辑 |

**实现要点**：题库生成 Agent 的 Prompt 从「基于岗位画像出题」改为「基于角色卡的技术栈和面试方向出题」。角色卡的 `question_categories` 字段控制各类题目数量配比。

#### 6.3.2 简历优化模块

| 维度 | V1.0（耦合） | V2.0（解耦） |
|------|-------------|-------------|
| 输入 | 简历文本 + JobProfile | 简历文本 + 活跃角色卡 |
| 调用方式 | `optimize_resume(resume, profile: JobProfile)` | `optimize_resume(resume, card: RoleCard)` |
| 兼容策略 | - | 角色卡的 core_skills 替代 JobProfile.core_skills 作为优化锚点 |

**实现要点**：简历优化 Agent 基于角色卡的 `tech_stack` 和 `core_skills` 判断简历中哪些技能描述需要强化、哪些技术栈需要补充提及。

#### 6.3.3 模拟面试模块

| 维度 | V1.0（耦合） | V2.0（解耦） |
|------|-------------|-------------|
| 输入 | JobProfile + 可选题库 | 活跃角色卡 + 可选题库 |
| 调用方式 | `create_session(id, profile, bank, resume)` | `create_session(id, card, bank, resume)` |
| 兼容策略 | - | 角色卡的 `interview_directions` 指导面试官出题方向 |

**实现要点**：模拟面试 Agent 的系统 Prompt 中注入角色卡的 `interview_directions`，使面试官问题更贴合角色方向。用户可选择是否先做题库再面试（有题库时面试官优先从题库选题）。

#### 6.3.4 项目梳理模块

项目梳理模块本身不依赖 JobProfile（V1.0 即独立），V2.0 保持不变，但可选注入角色卡上下文以使项目描述更贴合目标方向。

### 6.4 兼容性保障

为确保 V1.0 已有功能平滑过渡，采用以下兼容策略：

1. **JobProfile 向上兼容**：原有基于 JobProfile 的 API 保留，内部自动适配为角色卡调用
2. **适配器层**：在 Agent 入口增加 `RoleCardAdapter`，将 RoleCard 转换为各 Agent 需要的参数结构
3. **渐进迁移**：前端新功能使用角色卡 API，旧功能逐步迁移，不破坏现有体验

### 6.5 全流程编排保留

解耦后全流程仍可通过 LangGraph 编排，但入口从「JD 文本」变为「JD 文本 + 是否生成专属角色卡」：

```
START → job_matcher_node → generate_role_card_node（新增）
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
          resume_optimizer   question_bank   project_story
                    └───────────────┼───────────────┘
                                    ▼
                              END（面试异步触发）
```

## 12. 风险评估

### 12.1 技术风险

| 风险编号 | 风险描述 | 影响程度 | 发生概率 | 应对措施 |
|----------|----------|----------|----------|----------|
| R-T01 | **LLM 结构化输出不稳定**：`with_structured_output` 偶发返回不符合 Pydantic 模型的数据 | 高 | 中 | 增加 JSON Schema 校验和重试机制（最多重试 2 次）；失败时返回友好错误提示 |
| R-T02 | **LLM 响应延迟过高**：复杂 Prompt（如全流程）响应超过 30 秒 | 高 | 中 | 设置 30 秒超时；前端展示进度提示；全流程拆分为异步任务，前端轮询结果 |
| R-T03 | **LLM API 限流**：比赛演示期间并发请求触发限流 | 中 | 低 | 实现请求队列排队；对非关键请求做降级处理；准备多个 LLM API Key 轮换 |
| R-T04 | **面试会话内存泄漏**：当前面试 session 存于 Python 内存字典，未清理 | 中 | 高 | 增加会话过期机制（30 分钟无活动自动清理）；生产环境迁移至 Redis |
| R-T05 | **角色卡与 JobProfile 适配器兼容问题**：解耦后旧 API 调用路径可能字段缺失 | 中 | 中 | 编写适配器单元测试覆盖所有字段映射；保留 JobProfile 原始 API 路径并行运行 |
| R-T06 | **MySQL JSON 字段查询性能**：job_profile、messages 等 JSON 字段在数据量增大后查询变慢 | 低 | 低 | 对高频查询字段提取为独立列并建索引；JSON 字段仅用于展示不用于查询条件 |
| R-T07 | **前端状态管理复杂度**：角色卡全局状态需在多页面同步 | 中 | 中 | 使用 Pinia 集中管理活跃角色卡状态；页面切换时从 Store 读取而非组件局部状态 |

### 12.2 产品风险

| 风险编号 | 风险描述 | 影响程度 | 发生概率 | 应对措施 |

| R-P01 | **预设角色卡覆盖度不足**：8 个角色卡无法覆盖所有细分方向（如测试开发、安全工程师） | 中 | 中 | 预设卡覆盖主流方向；长尾方向通过专属角色卡（JD 生成）覆盖；V3.0 扩充预设卡 |
| R-P02 | **角色卡切换导致上下文丢失**：用户切换角色卡后，之前生成的题库/简历仍关联旧卡 | 中 | 中 | 历史记录保留关联的角色卡快照；切换时提示「历史记录不受影响」 |
| R-P03 | **模拟面试体验不够真实**：AI 面试官问题质量不稳定，可能过于简单或偏离方向 | 高 | 中 | 优化面试官 Prompt，增加出题约束；引入题库选题优先策略；面试阶段强制覆盖核心方向 |
| R-P04 | **简历优化建议过于笼统**：LLM 生成的建议缺乏具体性，用户感知不到价值 | 中 | 中 | Prompt 中要求引用简历原文和角色卡技能做对照；建议必须包含「原文 → 优化 → 理由」三段式 |
